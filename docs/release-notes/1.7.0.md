# Zocial 1.7.0

*Released 2026-06-10*

## ✨ New Features

**Translation: choose your target language**

Until now, Zocial picked your translation target automatically from your browser's language setting. That worked well for most people, but left no room for bilingual users or anyone whose browser speaks a different language than the one they actually want to read in.

1.7.0 adds a translation target language selector to Settings → General. The dropdown lists every language your Zocial instance's translation backend actually supports — not a hardcoded list, not a guess. When you open the settings page, Zocial quietly fetches the live language list from LibreTranslate in the background and populates the selector on the fly.

A few things make this smarter than it might look:

- **Per-instance cache:** If you're logged in to multiple servers, each one keeps its own language list. Switching accounts never shows you the wrong backend's capabilities.
- **Session deduplication:** The list is fetched at most once per session per instance. No repeat requests on every settings visit, no unnecessary backend load.
- **Resilient on failure:** If the request fails — rate limit, network blip, server error — the flag resets automatically so the next visit retries. An empty language list (valid: the backend returned successfully but supports no languages) is treated as a real result and not retried.
- **Graceful degradation:** If no language list is available, the selector is shown but disabled with a short hint explaining why. Your previous translation preference is preserved unless the chosen language is no longer offered by the backend, in which case it resets quietly to the browser default.

**Translation: better feedback on the way**

Two quality-of-life improvements that shipped in the 1.6.1 cycle and round out the translation experience:

- **Same-language posts are handled gracefully:** When the detected language of a post matches your target language, Zocial no longer surfaces an error — it shows a friendly "already in your language" message instead.
- **Error messages are now specific:** Rate-limit responses and unsupported-language errors each show their own tailored message rather than a generic failure notice.

**Lists: more resilient, less surprising**

- The lists feature had a silent but annoying flaw: any error during the list fetch — a temporary 429, a 500, a network hiccup — would mark lists as "unsupported" for the rest of the session and hide the entire lists UI. If you hit a rate limit at exactly the wrong moment, your lists just disappeared.
- Lists are now only hidden for definitive "not supported" responses (HTTP 403, 404, and 501 — the replies that actually mean "this server doesn't do lists"). Transient errors leave the lists UI intact and fall back to whatever is cached in the store. Your lists don't vanish because of a momentary hiccup.
- Alongside this: navigating to a list timeline now always triggers a fresh fetch, matching the behaviour of local and federated timelines and ensuring you see up-to-date content when you arrive.

**Timelines: 60-second polling fallback**

Zocial uses WebSocket streaming to keep timelines live wherever the server supports it. But some backends — Friendica, certain Akkoma configurations, and others — don't offer a streaming API at all. On those servers, timelines would only update when you manually refreshed.

1.7.0 adds a 60-second polling fallback that fires automatically when no streaming connection is active. The poll uses the same throttle as manual navigation (30 seconds of freshness grace, so a just-loaded timeline isn't redundantly re-fetched) and is gated precisely:

- It only runs while a Timeline component is actually mounted — no background fetches while you're in a different part of the app.
- It integrates with the existing `scheduleInterval` scheduler, which pauses automatically when the browser tab is inactive and resumes when you switch back.
- On servers that do offer streaming, the polling path remains in place as a safety net but typically resolves immediately against the warm cache.
- The net result: on non-streaming backends, timelines stay reasonably fresh with zero user action required.

## 🐛 Bug fixes

- **Unhandled promise rejections on Community settings:** Failures in `updateListsForInstance` and `updateFollowRequestCount` during the community settings load could produce silent unhandled rejections. Both are now caught and logged as warnings without interrupting the page.
- **"Create list" errors swallowed silently:** If creating a new list failed (rate limit, network error, etc.), the dialog closed and nothing happened. The error is now surfaced via toast notification so you know the action didn't go through, with the full error detail available in the console.
- **General reliability:** Several internal fixes to the translation fetch cycle, list fetch flag handling, and timeline observer setup to eliminate edge-case state inconsistencies.
