# Zocial 1.9.0

A major release that introduces a complete **notification system** and bundles a full **code-review hardening pass** across nine core subsystems — including a security fix to the login flow. (Everything since 1.8.2.)

## ✨ Features

### New notification system
- **OS notifications via Web Push** — rich, type-specific notifications that work even with the tab closed or as a mobile PWA. Each notification deep-links to the right place, and **mentions carry Boost / Favourite action buttons** you can use straight from the OS notification.
- **Honest single-account model.** Web Push is, by platform design, limited to one account per device. Zocial now makes this explicit instead of letting accounts race: enabling push on another account shows a **confirmation dialog**, cleanly moves push to it, and a permanent hint explains the limit. The active account is shown a clean state; others display "currently active for @X".
- **Correct multi-account routing.** Even with several accounts logged in, the service worker resolves each push to the right account (token → instance lookup), so the active account always gets the full rich notification rather than a bare fallback.
- **In-app notifications + notification sound**, fully independent of OS push, with their own filters and a dedicated sound toggle.
- **OS notifications are push-only.** The old foreground popup raised by the open page was removed; OS popups now come solely via Web Push. The master toggle ("Enable OS push notifications on this device") honestly reflects push state — it turns itself off if registration fails or the server doesn't support push. While the app is open and visible no duplicate OS popup appears (you just hear the sound and see it in-app).
- **Self-healing & honest failure state.** Push silently re-registers after a browser-side subscription loss; after repeated failures it gives up and turns the toggle off, so the switch never lies about being "on". The settings checkbox stays in sync with the real push state.
- **Hardened delivery.** Malformed, empty, or unknown-type push payloads still surface a notification rather than being silently dropped (a dropped `userVisibleOnly` push can otherwise get the subscription revoked).
- **Full localisation** of all new strings (English, German, Spanish, French, Russian).

### Follows & followers lists
- The **Follows** list now has a **follow/unfollow button** and the profile's **"⋯" menu** on each row, styled exactly like the profile.
- **Unfollow doesn't make the row vanish** — the button just flips to "Follow", so a misclick is easy to undo. Unfollowed accounts are gone the next time you open the list.
- The **Followers** list has the same button + "⋯" menu, so you can **follow back** directly: each button reflects your real relationship — "Follow" for people you don't follow yet, "Unfollow" for mutuals.
- On both lists the button reflects your real relationship, so it's also correct when viewing someone else's lists.
- Both the **Follows and Followers** lists now have the floating **scroll-to-top button** (like threads and profile timelines) once you've scrolled past the header.
- Empty account lists (followers, follows, blocked, muted, …) now explain **why** instead of showing a blank page: a load error, "nothing here yet", or — when the account reports a non-zero count but the list comes back empty — "this list isn't available here (kept private, or your server doesn't have it)". This covers the common Fediverse cases: a remote account whose social graph your server doesn't hold, an account that hides its followers/follows, or GoToSocial (which only exposes these lists for your own account).
- When a followers/follows list is **partial** — the account's count is higher than what loaded and the server offers no further page — a "Showing X of Y — your server doesn't provide the rest" note now appears, instead of a silently truncated list with no "Load more" button (the followers/following pagination cursor only exists in the server's response, so the client can't page further on its own).

### Timeline reliability
- **Cache-first rendering** on a cold load, for every normal timeline (home, local, federated, direct, notifications, account, list, tag): the last-seen feed appears instantly from local storage instead of a blank spinner, then refreshes from the network.
- **List & hashtag timelines** are tuned for slow servers (longer read timeout, smaller first page) and now **auto-retry once** on a transient first-load timeout — so a slow server-side list assembly recovers in seconds instead of needing a manual page refresh. Already-visible content is never disturbed.

## 🔒 Security

- **OAuth `state` CSRF protection (RFC 6749 §10.12).** The login flow now generates a random, unguessable `state` token before redirecting to your instance, and verifies it on the callback **before** exchanging the authorization code. This closes the standard OAuth login-CSRF vector — a forged or replayed callback that wasn't started by your own login is rejected. (Login + logout verified.)

## 🐛 Fixes & hardening (code-review pass)

A focused review of nine neuralgic subsystems; highlights:

- **Compose** — closed a data-loss window in delete-and-redraft (the dialog and reply context are now resolved *before* the post is deleted); poll-only posts are allowed and empty/invalid posts give feedback instead of a dead click; a failed media-metadata update no longer aborts the whole post; per-box text autosave (no cross-box overwrite); cached media blobs are cleaned up on removal.
- **Timeline read** — fixed a crash that could blank the timeline when a stored pointer outlived its post; list/tag cold-load auto-retry (above).
- **Streaming** — unified the reconnect path (no more double-connect churn or tearing down a healthy socket on a spurious "online" event); malformed websocket frames can no longer throw out of the handler.
- **IndexedDB** — concurrent first-time database opens are de-duplicated, fixing leaked connections on cold load that could later block logout.
- **Service worker** — mention action buttons now post to the correct instance API host (previously derived from the avatar URL, which breaks on CDN-hosted avatars).
- **Word filters** — an empty/malformed filter phrase can no longer turn into a catch-all that silently hides your entire timeline.
- **Polls** — a poll whose payload omits `own_votes` or `options` no longer throws inside a computed and cascades into a broken/blank timeline. This can happen on any backend (Mastodon included) for remote polls, streamed updates, or cache-first renders from local storage; both fields are now guarded.
- **Auth/logout** — logout now clears the correct account's view cache.
- **Virtual list** — scroll handler is cancelled on teardown (no stray writes).

## 🧹 Internal / quality

- Removed noisy debug logging across hot paths (timeline, streaming, compose, DB, filters); the robust in-app log viewer (with global uncaught-error/rejection capture, persisted across reloads) is unchanged.
- Architecture handbook updated: new **§22 Code Review Log** documenting every review pass and outcome, plus reconciled notification (§18) and design-decision (§20) sections.

## ⚠️ Known limitations

- **OS push is one account per device** — a hard Web Push platform limit (one VAPID-keyed subscription per origin). Other logged-in accounts notify you in-app; true cross-instance background push would require a self-hosted relay (out of scope).
- Clearing an account's local database can stall if another browser tab holds an open connection to it (noted for a future pass).
